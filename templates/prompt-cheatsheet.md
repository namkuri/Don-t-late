# 프롬프트 치트시트 (Prompt Cheat Sheet)
## 상황별 좋은 예시 모음 — 심야배송 / Unity / 에이전트 파이프라인

> 개발 중 애매한 프롬프트가 나올 때 **여기 보고 베낀다.** 이론 없음, 예시만. (출처: Anthropic 공식 프롬프트 가이드를 이 프로젝트 맥락으로 각색)

---

## 0. 만능 원칙 (다 잊으면 이 5개만)

1. **황금률** — 맥락 없는 동료가 이 프롬프트로 실행 가능한가? 아니면 애매한 거다.
2. **5칸 계약** — `목표 · 입력 · 기대 산출물 · 수용 기준 · 실패 시`. 지시는 이 틀로.
3. **구체 + 왜** — 무엇을 원하는지 + 왜 그런지. 이유를 주면 정확도가 오른다.
4. **하지 마 → 해라** — "마크다운 쓰지 마" ✗ → "산문으로 써" ✓. 긍정 지시로.
5. **XML로 분리** — 지시·입력·기준이 섞이면 `<목표>`·`<입력>`·`<기준>`으로 감싼다.

## 1. 애매 → 구체 리라이트 (빠른 참조)

| ✗ 애매 | ✓ 구체 |
|---|---|
| "문 열어" | "엘베 문을 X축 +0.9m 슬라이드, 0.4초, ease-out. E키 입력 시 토글" |
| "포터 트럭 만들어" | "무개 포터 트럭, 저폴리 <1500, PS1 톤, GLB, 뚜껑 있으면 반려" |
| "메모 좀 만들어줘" | "진상 배송 메모 20종, 각 30자 이내, `data/schemas/memo.json` 스키마 준수" |
| "이거 검수해" | "이 프리팹을 CLAUDE.md 수용 기준(폴리·임포트에러·인터페이스)과 대조, ACCEPT/REJECT+사유 반환" |
| "빠르게 해줘" | "이 3개 파일은 서로 독립적이니 병렬로 읽어라" |
| "코드 개선해" | "이 함수의 null 참조 버그만 고쳐라. 다른 리팩터·주석 추가 금지" |

## 2. 3D 에셋 생성 (Meshy / Trellis)

```
목표    : 단지 소품 "우편함" 생성
지시    : low-poly PS1-style mailbox, worn metal, 저해상 텍스처,
          폴리 < 1000, game-ready GLB
입력    : (선택) 컨셉 원화 mailbox_concept.png
기대    : assets/props/mailbox.glb + 매니페스트 기록
수용기준: 폴리<1000, Unity 임포트 에러 0, PS1 톤 일치
실패 시 : 폴리 초과/톤 이탈이면 프롬프트 수정해 1회 재생성, 2회 실패 시 보고
```

> **배치 생성 팁** — 비슷한 소품은 묶어서: "택배상자 대/중/소 3종을 한 번에 생성". 크레딧 절약.

## 3. Unity 로직 구현 (구체 수치가 생명)

```
목표    : 스태미나 시스템 구현
지시    : 계단 1층 오를 때 -8, 엘베 탑승 시 매초 +5(최대 100),
          0 되면 이동속도 50% 감속. StaminaManager 단일책임으로,
          다른 매니저와는 이벤트로만 통신(직접 참조 금지).
기대    : StaminaManager.cs + 씬 반영
수용기준: 컴파일 통과, 콘솔 에러 0, Play모드에서 감소·회복 확인
실패 시 : 2회 실패 시 사람 게이트
```

> Play모드 정지 상태에서만 씬 구조를 바꾸라고 명시.

## 4. 콘텐츠 배치 생성 (메모 · 후기 → JSON)

```
목표    : 배송 후기 30종 생성
지시    : 다크 코미디 톤, 별점 1~5 분포, 각 40자 이내.
          출력은 오직 JSON, 서두·설명·마크다운 없이.
          data/schemas/review.json 스키마 정확히 준수.
기대    : reviews.json (30개 항목)
수용기준: 스키마 검증 통과, 중복 없음, 톤 일치
```

> **"오직 JSON, 서두 없이"** 를 꼭 박는다. 안 그러면 앞뒤에 설명이 붙어 파싱이 깨진다.

## 5. 검수 요청 (reviewer)

```
너는 품질 게이트다. 만들지 말고 검사만 한다.
1. CLAUDE.md 수용 기준을 로드한다.
2. 대상 산출물을 기준과 대조한다.
3. 반환: ACCEPT  또는  REJECT + 사유 + 구체적 수정 지시.
기준 미달을 관대하게 넘기지 마라. 반려 사유는 로그에 남길 수 있게 구체적으로.
```

---

## 6. 서브에이전트 위임 (언제 / 언제 안)

```
서브에이전트는 이럴 때 써라: 병렬 실행 가능 · 독립 컨텍스트 필요 ·
상태 공유 불필요한 별개 작업.
직접 처리해라: 단순 작업 · 순차 작업 · 단일 파일 편집 ·
단계 간 맥락 유지가 필요한 작업.
```

> Opus는 서브에이전트를 과하게 쓰는 경향 → "간단하면 직접 해라"를 명시하면 억제됨.

---

## 7. 도구 실행 확실히 (제안 말고 실행)

```
기본적으로 제안이 아니라 실제 구현/실행을 해라.
의도가 불명확하면, 가장 유용할 행동을 추론해 진행하되,
빠진 정보는 추측하지 말고 도구로 확인해라.
```

> "제안해줄래?"라고 하면 실행 안 하고 제안만 하는 경우가 있음 → 명시적으로.

---

## 8. 병렬 도구 호출 (속도)

```
서로 의존 없는 도구 호출은 병렬로 해라. 파일 3개를 읽어야 하면
3개를 동시에 호출해라. 단, 한 호출의 결과가 다른 호출의 인자로
필요하면 순차로 해라. 파라미터를 추측하거나 플레이스홀더로 채우지 마라.
```

---

## 9. 출력 간결 / 형식 제어

```
# 장황함 억제
불필요한 요약·서두 생략. 도구 실행 후 바로 다음 작업으로.

# 진행 요약이 필요할 때만
도구 사용을 포함한 작업 완료 후, 한 일을 짧게 요약해라.

# 형식은 긍정으로
"마크다운 쓰지 마" ✗ → "완결된 문장의 산문 단락으로 써라" ✓
```

## 10. 과잉 엔지니어링 방지 (스코프 고정) ⭐

```
과잉 설계 금지. 직접 요청되었거나 명백히 필요한 변경만 해라.
- 범위: 요청 안 된 기능·리팩터·"개선"을 덧붙이지 마라. 버그 수정은
  주변 코드 정리를 포함하지 않는다.
- 방어 코딩: 일어날 수 없는 상황의 예외처리·폴백·검증을 넣지 마라.
  시스템 경계(사용자 입력·외부 API)에서만 검증해라.
- 추상화: 일회성 작업에 헬퍼·유틸을 만들지 마라. 가상의 미래
  요구를 위해 설계하지 마라. 지금 작업에 필요한 최소 복잡도만.
```

> **정제 에이전트에도 이걸 박는다** — "명확화만, 범위 확장 금지."

---

## 11. 코드 환각 방지 (파일 먼저 읽기)

```
열어보지 않은 코드에 대해 추측하지 마라. 특정 파일이 언급되면
답하기 전에 반드시 그 파일을 읽어라. 코드베이스 질문 전에 관련
파일을 조사·확인하고, 확신이 없으면 단정하지 마라.
```

---

## 12. 위험 행동 안전장치 (되돌리기 비싼 것) ⭐

```
행동의 되돌림 가능성과 영향을 고려해라. 파일 편집·테스트 실행 같은
로컬·가역 작업은 자유롭게. 하지만 되돌리기 어렵거나 공유 시스템에
영향을 주거나 파괴적인 작업은 먼저 사람에게 확인해라.
확인이 필요한 예: 파일/브랜치 삭제, git push --force, 메인 커밋,
외부 게시. 막힐 때 파괴적 행동을 지름길로 쓰지 마라(--no-verify 등 금지).
```

> AAPP의 "cast 발주·커밋·삭제 = 사람 게이트"와 정확히 연결.

## 13. 장기 작업 상태 추적 (여러 컨텍스트 윈도우)

```
- 진행 상태를 구조화 형식으로: tests.json, TASKS.md 에 기록.
- git을 상태 추적에 써라. 체크포인트마다 커밋.
- 새 컨텍스트 시작 시: pwd 확인 → progress·TASKS·git log 검토 →
  이어서 진행. 토큰 부족으로 작업을 임의로 조기 종료하지 마라.
- 컨텍스트 한계 근처면 진행 상태를 메모리에 저장 후 계속.
```

---

## 14. 자체 수정 체이닝 (초안 → 검토 → 개선)

```
1) 초안 생성
2) [기준]에 따라 스스로 검토 — 무엇이 미달인지 짚어라
3) 검토를 반영해 개선
완료 전, [수용 기준]에 대해 답을 검증해라.
```

> 코딩·데이터에서 오류를 안정적으로 잡는 패턴. 정제 에이전트의 뼈대이기도.

## 15. 프론트엔드(웹빌드 UI) — "AI slop" 회피

```
일반적이고 뻔한 출력으로 수렴하지 마라("AI slop" 미학 금지).
- 타이포: Inter·Roboto·Arial 같은 뻔한 폰트 회피, 개성 있는 선택.
- 색: 하나의 응집된 미학에 커밋. CSS 변수로 일관성. 지배색 +
  날카로운 강조색이 밋밋한 균등 팔레트를 이긴다.
- 모션: 고임팩트 순간에 집중(스태거드 페이지 로드 하나가
  산발적 마이크로 인터랙션보다 낫다).
- 배경: 단색 대신 그라디언트·패턴·분위기로 깊이.
```

> 심야배송이면 PS1 CRT·나트륨등 야간 톤으로 방향을 이미 잡아뒀으니 그걸 명시.

## 사용 규칙 (요약)

- **매번 정제하지 마라.** 재시도 났거나 · 되돌리기 비싼 작업 · 준비 기간 반복 작업일 때만.
- **정제 결과는 `prompts/`에 저장** → prompt_ref로 재사용(스킬 축적).
- **본선엔 이 치트시트 + 저장된 prompt_ref를 재생만.** 실시간 정제 최소화.
