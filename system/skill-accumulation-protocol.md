# 스킬 축적 프로토콜 (Skill Accumulation Protocol)
## 준비 기간용 저비용 자기개선

> **한 줄 정의** — 준비 기간(25일)에 파이프라인을 돌리며 **잘 된 것을 스킬 문서에 append**로 굳혀, 본선엔 *학습이 아니라 재생*만 하게 만든다. Hermes 같은 자기개선 런타임 없이, **트래젝토리→스킬**의 핵심만 사람이 수동으로 취한다.

- **소속**: `finals-execution-system-v2.md`의 준비 계층 (부록 B 레일 구축의 운영 절차)
- **성격**: 무거운 시스템이 아니라 **가벼운 규율.** 로그 한 줄 + 복기 몇 분이 전부.

---

## 0. 원리

- **준비에서 배우고, 본선에서 재생한다.** 자기개선은 반복이 있어야 도는데, 반복은 25일 준비에만 있다. 본선 48h는 각 작업이 대체로 1회라 학습이 안 쌓인다.
- **측정이 연료다.** 자기개선은 측정을 없애지 않는다 — "잘됨/못됨" 신호가 있어야 스킬로 굳힐 수 있다. 이 프로토콜은 그 측정을 **복기에 흡수**한다(§7).
- **우리는 이미 스킬 기반이다.** CLAUDE.md · `.claude/agents/` · AAPP 공정 시트가 곧 스킬 문서다. 여기에 "잘 된 걸 계속 append"하는 게 저비용 자기개선이다.
- **Append-only.** 스킬은 수정이 아니라 추가로 자란다(§8, OCP). 정리는 정기 복기 때 몰아서.

---

## 1. 스킬이 사는 곳

증류된 스킬은 성격에 따라 정해진 문서로 간다.

| 스킬 종류 | 착지 문서 |
|---|---|
| 범용 규칙·컨벤션 | `CLAUDE.md` |
| 역할 프롬프트 개선 | `.claude/agents/*.md` |
| op 공정·순서·모델 배정 | AAPP 표준 공정 시트 |
| 검증된 프롬프트 | `prompts/*.md` (prompt_ref로 참조) |
| op×model 성능 데이터 | `calibration.md` |
| 하지 말 것 | `ANTIPATTERNS.md` |

---

## 2. 작업 세트 — 세 개의 파일만

무겁게 만들지 않는다. 실제로 다루는 건 셋뿐이다.

1. **트래젝토리 로그** (`logs/trajectory.md`) — 매 실행의 관찰 기록 (§4).
2. **복기 노트** (`logs/retro.md`) — 복기에서 나온 결정 (§5).
3. **스킬 문서** (§1의 착지 문서들) — 승격된 스킬 (§6).

흐름: **로그(관찰) → 복기(판단) → 스킬(결정화).**

---

## 3. 핵심 루프

```
[실행] 파이프라인 한 번 돌린다
   │
[로그] 트래젝토리 한 줄 기록 (op·model·성공여부·재시도·한줄노트)
   │
[복기] 잘된 것/재시도한 것/실패한 것/낭비한 것을 본다   ← "중간중간 대화 복기"
   │
[증류] 반복·우위·일반성 만족하면 스킬로 승격
   │
[append] 해당 스킬 문서에 추가
   │
[재생] 다음 실행은 그 스킬을 불러 씀 → 점점 나아짐
```

---

## 4. 트래젝토리 로그 (한 줄이면 충분)

실행마다 아래를 **한 줄** 남긴다. 정밀할 필요 없다. 핵심은 성공/실패와 재시도 횟수.

```yaml
- run: 07-20 #3
  op: implement-logic
  task: "엘베 문 슬라이드"
  model: sonnet
  prompt_ref: P-014
  outcome: pass-after-retry   # pass | fail | pass-after-retry
  retries: 1
  note: "X축 명시하니 1회 교정 통과. '문 열어'는 실패했었음"
```

> 가장 중요한 필드는 `outcome`과 `retries`. 이 둘이 품질의 실용 프록시다(1패스 통과 = 좋음, 재시도 잦음 = 나쁨).

---

## 5. 복기 (Retrospective) — 두 리듬

"중간중간 복기해서 업그레이드"의 실체. 두 주기로 돈다.

### 5.1 실행 직후 (매 run · 약 2분)

로그를 보며 4가지만 확인:

- [ ] **뭐가 1패스로 통과?** → 그 프롬프트/공정 = **승격 후보**
- [ ] **뭐가 재시도했나?** → 왜? 프롬프트를 고칠까, 모델을 올릴까?
- [ ] **뭐가 반복 실패?** → 스킬로 "이건 이렇게" 굳히거나, 이 op는 사람이 하기로 결정
- [ ] **낭비된 시도 있나?** → "이건 하지 마"를 `ANTIPATTERNS.md`에 기록

### 5.2 정기 복기 (하루 끝 or N회마다 · 약 10분)

- [ ] 로그에서 **2회 이상 반복된 성공 패턴** → 스킬 승격(§6)
- [ ] op×model **성공률 집계 갱신** → `calibration.md` 업데이트(§7)
- [ ] 스킬 문서 **비대화 점검** → 중복·낡은 것 정리·통합(§8)
- [ ] 안티패턴 목록 갱신

> **대화 복기 = 트래젝토리 분석.** 실제 세션 기록에서 "잘 통한 프롬프트·교정·순서"와 "막다른 길·낭비 재시도"를 뽑아 각각 스킬/안티패턴으로 옮기는 것.

---

## 6. 승격 규칙 — 언제 노트가 스킬이 되나

아무거나 스킬로 굳히면 문서만 비대해진다. **3조건을 다 만족**할 때만 승격.

| 조건 | 판정 |
|---|---|
| **반복성** | 같은 패턴이 2회 이상 성공했나? |
| **우위** | 대안보다 측정상 나은가? (성공률↑ 또는 실효비용↓) |
| **일반성** | 다른 태스크에도 적용되나? |

- 3개 다 만족 → **스킬로 승격**, 해당 문서에 append.
- 하나라도 미달 → 로그 노트로만 남김(아직 스킬 아님).

---

## 7. 모델 캘리브레이션 흡수

앞서 미룬 모델 측정은 **별도 활동이 아니라 복기의 부산물**이다. 트래젝토리 로그에 이미 op·model·outcome·retries가 있으니, 정기 복기 때 집계만 하면 캘리브레이션 표가 자란다.

```yaml
# calibration.md (복기가 갱신하는 스킬 문서)
- op: write-manifest
  haiku:  {pass_rate: 0.95, eff_cost: 1.0}   # 실효비용 = 1회비용 ÷ 성공률
  sonnet: {pass_rate: 0.99, eff_cost: 4.0}
  decision: haiku            # 품질 임계 넘는 것 중 실효비용 최저
- op: implement-logic
  haiku:  {pass_rate: 0.40}  # 재시도 폭주 → 탈락
  sonnet: {pass_rate: 0.75}
  opus:   {pass_rate: 0.92}
  decision: opus
  escalation: "haiku→sonnet→opus (2회 실패 시 상향)"
```

> 결정 규칙: **품질 임계(성공률)를 넘는 모델 중 실효 비용이 가장 싼 것.** 이 표가 AAPP 공정 시트의 `model`·`escalation` 필드를 채운다. 감이 아니라 데이터로.

## 8. Append & 정리 (OCP 규율)

- **실행 중엔 append만.** 잘 된 걸 계속 추가. 기존 스킬을 실시간으로 고치지 않는다(흔들림 방지).
- **정리는 정기 복기 때 몰아서.** 중복·낡은 스킬을 통합·삭제. append-only의 비대화를 여기서 해소.
- 스킬 문서도 §12 변경관리의 대상 — 준비 기간엔 유연, 본선 직전 **동결**.

---

## 9. 본선 모드 — 학습 끄고 재생만

준비 기간이 끝나면 스킬을 **동결**한다.

- 본선 48h엔 트래젝토리 축적·스킬 승격을 **하지 않는다.** 굳어진 스킬을 불러 재생만.
- 예외: 본선 중 명백한 안티패턴을 만나면 **회피**만 한다(그 자리에서 스킬을 개정하진 않음 — 시간 낭비).
- 이유: 본선은 반복이 적어 학습이 안 쌓이고, 실행 중 자기개선은 통제 흔들림·시간 소모를 부른다.

---

## 10. 최소 실천 (쉬운 길 요약)

무겁게 느껴지면 이것만 해도 90%다.

- **로그는 한 줄** — `op · model · pass/fail · retries · 한줄노트`.
- **복기는 매 run 2분 + 하루 끝 10분.**
- **승격은 한 규칙** — "2번 잘됐고 대안보다 나으면 굳힌다."
- **캘리브레이션은 자동** — 로그 집계가 곧 측정. 따로 벤치마크 안 돌려도 됨.
- **본선엔 재생만** — 굳힌 스킬 불러 쓰고, 학습은 끈다.
