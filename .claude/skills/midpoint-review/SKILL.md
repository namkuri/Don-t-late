# midpoint-review — 하네스 중간점검 감사

> 원 프롬프트(2026-07-21 남규)를 정형화한 스킬. 개선점: ① 근거 소스 고정 ② 검증 등급 정의
> ③ "사람이 짚은 것" 전수 분석을 필수 섹션으로 ④ 관제 자신의 위반을 반드시 포함 ⑤ 델타 모드.

## 산출물

`planning/retrospective-YYYY-MM-DD.md` 1개. **이전 회고 파일이 있으면 "전 회고 백로그 이행 여부"
델타 섹션을 맨 앞에** 추가한다 (백로그가 소화 안 됐으면 그게 1번 지적사항이다).

## 절차

### 1. 근거 수집 (추측 금지 — 전부 실측)
- `git log --oneline` (커밋 수·경계 준수), [[TASKS]] 상태 분포, [[decisions]] D-번호 범위,
  [[iterations]] 반려 건수, [[ai_evidence]] 발주 건수, [[INBOX]] 적체 현황
- 에이전트 usage(소요시간·토큰)가 있으면 eta 집계에 반영
- 규칙 파일의 "실수→규칙" append 수 (`grep -c`)

### 2. 타임라인 표
구간 / 일어난 일 / **하네스 관점 의미** 3열. 사건 나열이 아니라 "레일이 뭘 배웠나"를 적는다.

### 3. 파트별 검증 매트릭스
등급 정의 고정: **L0** 미검증 · **L1** 컴파일/콘솔 · **L2** 기계 실측(수치·샘플) ·
**L3** 관제 육안(스크린샷) · **L4** 사람 실플레이/판정.
열: 파트 / 상태 / 등급 / 근거 / **잔여 리스크**. done인데 L2 이하인 항목은 🔴 표시
(예: "WebGL done"이 구버전 빌드 기준이면 그것이 최대 구멍이다 — 상태와 검증의 불일치를 찾는 게 핵심).

### 4. 하네스 자가 개선 대장 (전수)
형식 고정: **문제 / 원인 / 개선(자동=구조·훅 | 수동=문서 규칙) / 결과(재발 여부) / 근거**.
말미에 원인 귀속 통계 — 에이전트 실수 vs **관제 봉투 결함** vs 외부 요인. 관제 결함이 다수면
그렇게 쓴다.

### 5. 사람이 짚었고 관제가 놓친 것 (전수 — 이 스킬의 심장)
사용자 메시지를 되짚어 "관제가 먼저 못 잡고 사람이 요구한 것"을 빠짐없이 표로:
지적 / 왜 놓쳤나 / 개선 / 결과 / 근거. 마지막에 **패턴 진단 한 단락** — 개별 사건이 아니라
공통 원인(예: 검증축이 기계 기준에 갇힘, 사람 인터페이스 부재)을 지목한다.

### 6. 날카로운 평가
- 금지: 아첨 · "문제없습니다" · 잘된 것만 나열
- 필수: 미검증은 미검증이라 명시 · **관제 자신의 규율 위반 포함**(국면 이탈·게이트 미집행·
  집계 태만 등 — question-gates 규율과 동일: 결함 최소 3개, 근거와 함께)
- 각 지적은 "무엇을 하면 해소되는지"까지 (지적만 하고 끝내지 않는다)

### 7. 개선 백로그
우선순위순 번호 목록. 각 항목은 다음 체크포인트에서 이행 검증 가능한 형태로
("~하기"가 아니라 "~1회 실행·결과 기록").

### 8. 총평 1단락
레일 상태를 은유 없이 한 단락. 다음 회고의 델타 기준점이 된다.

## 운용 규칙
- **저맥락 링크 의무 (D-023 — 회고 문서도 예외 없음)**: ID(B-x·D-xxx·M1-xx·Q-x·§번호)는 첫 언급에
  Foam 링크 + 한 줄 평문 설명. 근거 열은 "사람이 열어볼 수 있는 것"(문서 링크·스크린샷 경로·코드
  주석 위치)으로 쓴다 — 내부 코드명·발주명 맨몸 표기 금지. (2026-07-21 실제 반려 사례: 회고 초판이
  이 규칙을 어겨 사람이 재작업을 요구함 — §5 표 13번)
- INBOX·STATUS에 회고 링크 등재. 백로그 중 사람 결정이 필요한 건 INBOX로 이관.
- 회고 자체는 판정이 아니다 — done 전이는 여전히 판정 주체(기계·reviewer·사람)만.
- 권장 주기: 이정표 전환 시(M1→M2 등) 또는 사람 요청 시. 상시 금지(소음).
