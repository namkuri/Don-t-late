# HARNESS.md — 하네스 헌장 (48h 본선판)

> **하네스 = 모델이 아닌 모든 것.** 이 킷 전체(CLAUDE.md·agents·pipelines·게이트·훅·로그)가 하네스다.
> 핵심 철학: **에이전트가 실수하면 프롬프트를 고치는 게 아니라, 그 실수가 구조적으로
> 반복 불가능하게 하네스를 고친다.** 프롬프트는 부탁이고, 하네스는 강제다.

## 0. 48h 제약 3원칙 (일반 하네스론과 다른 점)

1. **하네스는 레일이다 — 본선 중 신축 공사 금지.**
   하네스 구축은 준비 기간(25일)의 일이다. 본선 48h 중 허용되는 하네스 변경은
   **"실수 1건 = 규칙 1줄 append"** 뿐이다(§3). 구조 개편·새 훅 제작·새 파이프라인 설계는 금지
   — 그건 게임 만들 시간을 먹는다. (변경관리 §12의 append-only와 동일 규율)
2. **자동교정은 retry_cap 안에서.**
   실패 시 스스로 고치되(§2) **2회 안에** 못 고치면 모델 상향 또는 사람. 무한 자기교정 루프는
   1인 대회의 사망 원인(시간·토큰 소모)이다.
3. **가비지 컬렉션은 상시가 아니라 체크포인트에서.**
   청소 에이전트를 상시 돌리지 않는다. 4시간 통합 체크포인트에 청소 항목(§4)을 포함해
   몰아서 처리한다.

## 1. 3기둥 매핑 (일반론 → 우리 킷)

| 하네스 3기둥 | 우리 킷의 구현 |
|---|---|
| 컨텍스트 파일 | CLAUDE.md(공통 룰) + docs/(INTENT·규격) + pipelines/(영역별). **지도 원칙**: 짧게, 보편 규칙만. 세부는 파이프라인 파일로 분산, 필요할 때만 로드 |
| 자동 강제 | 게이트(reviewer·사람 승인) + 훅(freeze-guard·프리커밋) + **자동교정 루프**(§2) |
| 가비지 컬렉션 | 체크포인트 청소(§4) |

## 2. 자동교정 루프 (모든 에이전트 공통)

```
산출 → [게이트 검사] → 실패? → 실패 원인 읽기 → 스스로 교정 → 재검사
                          │ (cap 2회)
                          └ 2회 실패 → 모델 상향(haiku→sonnet→opus) or 사람 게이트
```
- 검사 기준은 각 pipelines/*.md의 게이트 정의를 따른다.
- **성공은 조용히, 실패만 시끄럽게**: 통과 시 1줄 요약만 반환(전체 로그 금지 — 읽느라
  컨텍스트 낭비). 실패 시에만 사유·수치·시도 내역을 상세히.

## 3. 실수 → 규칙 프로토콜 (하네스가 자라는 방식)

- **실수 1건 = 해당 pipelines/*.md 의 "## 실수→규칙" 섹션에 1줄 append.**
- 형식: `- [국면N] <증상> → <금지/강제 규칙>`
- 누가: **실수를 낸 에이전트가 직접** append (교정 완료 직후, 잊기 전에).
- 검수: 사람이 체크포인트에서 훑고, 잘못 굳은 규칙은 사람만 제거 가능.
- 처음부터 완벽한 규칙집을 만들지 않는다 — **실패할 때마다 한 줄씩** 자란다.
- 같은 실수가 규칙 추가 후에도 반복되면: 규칙(부탁)이 아니라 **구조(게이트·훅)**가 필요하다는
  신호 → 사람에게 보고 (본선 중 구조 공사는 사람이 결정).

## 4. 가비지 컬렉션 체크리스트 (4h 체크포인트마다, orchestrator 주관)

- [ ] 문서-실물 드리프트: BOM/TASKS 상태가 실제 씬·파일과 일치하나
- [ ] 미검증 커밋·태그 누락 커밋 없나
- [ ] 죽은 에셋: placeholder 교체 후 방치된 그레이박스, dest 밖 떠도는 파일
- [ ] 규칙 위반 잔재: 매니저 직접 참조, 하이폴리 잔존, 매니페스트 누락
- [ ] logs/ 최신성: ai_evidence·iterations가 실제 작업과 동기화됐나
- [ ] **_intake 잔류물**: 도구별 폴더에 검역 미통과 방치물 없나 (적체 = 병목 신호, §9)

## 5. 연결 관리 (외부 도구 하네스)

- 외부 도구 현황은 **CONNECTIONS.md**(연결 대기판)가 단일 진실.
- 모든 에이전트는 작업 시작 전 자기 파이프라인의 필요 연결을 점검한다.
- **미연결 발견 시 침묵 실패·추측 진행 금지.** 아래 형식으로 사람에게 역요청:

```
[CONNECT_REQUEST]
도구      : <이름>
막힌 작업 : <bom_id / task — 무엇이 진행 불가인가>
연결 방법 : <사람이 할 일 — MCP 등록·API키·수동 절차>
폴백      : <연결 불가 시 대안과 그 비용>
긴급도    : 치명(레인 전체 정지) | 높음(해당 작업군 정지) | 보통(폴백 존재)
```
- orchestrator가 수신 → CONNECTIONS.md 상태 갱신 → 사람 에스컬레이션 → 해당 태스크는
  `blocked(원인 명시)`. 폴백이 있으면 폴백 전환을 함께 제안한다.

## 6. 판정 권한 — 작성자 ≠ 판정자

자기평가 편향: **만든 자가 판정하면 반드시 후하다.** 작업한 세션에서 "네 작업 평가해봐"는 금지 구조다.
- done 판정 주체는 셋뿐: ⓐ 기계 기준(컴파일·스키마·수치·훅) ⓑ 작성자와 **다른 모델**의 reviewer ⓒ 사람.
- 작성 에이전트의 셀프체크·자기 보고 = **"제출"이지 "판정"이 아니다** (reviewer에 낼 자격 필터일 뿐).
- TASKS 전이: `doing→review`는 작성자가 올리고, `review→done`은 판정 주체만.
- 보고 언어: 판정어("완료했습니다") 금지 → 재현 가능한 관찰 기록("E키 입력 시 0.4초 슬라이드 관찰됨").

## 7. 도구 경로 고정 & 에러 원문

- **지정 도구가 있는 작업은 그 도구로만.** 에디터 조작=unity-cli — ⓐ `exec`(에디터 API) 우선,
  ⓑ 씬·프리팹 YAML 텍스트 편집은 **`reserialize` 동반 시에만 허용**(재직렬화 없는 YAML 편집=금지, 무효 에셋 위험).
  외부 에셋=지정 생성 도구만(웹 임의 다운로드 금지 — 라이선스 리스크).
- 도구가 미연결·실패라고 **대체 경로로 새지 않는다** → CONNECT_REQUEST가 정답이다.
- **도구 에러는 요약·삼킴 금지** — 원문 핵심부를 부모·로그에 그대로 전달한다. "실패만 시끄럽게"의 정의 = 원문 포함. (쿼터 소진·인증 만료 신호는 원문에만 있다 — CONNECT_REQUEST 긴급도 판단 근거.)

## 8. 훅 제작 목록 (준비기간 레일 공사 — 본선 중 신축 금지)

NEVER 목록 감사 결과: **기계가 판별 가능한 규칙은 훅으로 승격**(문서=부탁, 훅=강제), 판단이 필요한 것만 문서+사람 게이트로 유지.

| 규칙 | 훅 (어기는 순간 자동 차단) |
|---|---|
| 미검증물 커밋 금지 | pre-commit: 컴파일+콘솔 검사, 실패 시 커밋 거부 |
| 동결문서 수정 금지 | freeze-guard: 기존 줄 diff 검출 시 저장 차단 (추가만 통과) |
| 라이선스 미기록 반입 금지 | 커밋 내 assets/ 신규 파일 ↔ manifest 대조, 기록 없으면 커밋 거부 |
| dest 밖 저장 금지 | BOM dest ↔ 실제 경로 대조 스크립트 (체크포인트 GC에서 실행) |
| 커밋 태그 누락 | commit-msg 훅: 태그 없으면 경고 |

훅화 불가(의미 판단 필요 — 문서+사람 게이트 유지): 조명 AI 금지 · 스코프 밖 기능 · 원프롬프트 생성기 · cast 조기 발주.

**채점기(verifier) 제작 목록** — 훅이 "금지의 강제"라면 채점기는 "판정의 자동화"다.
사람 눈을 기계로 대체할수록 본선의 사람 시간이 돌아온다. **시스템 성능 = 채점기 커버리지.**

| 채점기 | 대체하는 사람 판정 |
|---|---|
| 팔레트 히스토그램 검사 (스크린샷 색분포 ↔ STYLE 팔레트 대조) | 톤 드리프트 육안 검사의 절반 |
| 체크포인트 스크린샷 자동 번들 (`unity-cli screenshot` 공급) | 씬 순회 → 넘겨보기로 단축 |
| 씬 통계 리포트 (총 폴리·드로우콜·텍스처 메모리 집계) | WebGL 예산 수동 확인 |

한계를 정직히: 채점기는 **신호기지 최종 판정이 아니다.** 플래그가 뜨면 사람이 본다
(거짓 양성 허용, 거짓 음성 최소화 방향). 그래도 "언제 봐야 하는지"를 기계가 알려주는 것만으로 예산이 준다.

## 9. INTAKE 프로토콜 — 외부 반입물의 단일 관문

모든 외부 산출물(생성 API·라이브러리 다운로드·사람 매개 반환물)은 출처 불문 이 관문을 통과한다.
리스크 4종(라이선스 누락=실격 사유 · 미검증 오염 · 재현 불가 · 잔해 방치)을 한 구조로 막는다.

```
assets/_intake/<도구명>/     ← ① 착지 격리 (dest 직행 금지)
        ↓ ② manifest 기록 = 입장권 (출처·프롬프트·라이선스·gen)
        ↓ ③ 검역 = reviewer 게이트 + 확장자 화이트리스트
assets/<dest>/<bom_id>.*     ← ④ 이동 (파일명 = bom_id 고정, 재생성=덮어쓰기)
        ↓ ⑤ _intake 원본 폐기 (재현 정보는 manifest가 보유)
```

- **① 착지 격리**: 도구명 하위 폴더에 착지 — `meshy/ trellis/ nanobanana/ blender/ mixamo/
  cascadeur/ suno/ elevenlabs/ freesound/ web/`. 폴더명은 CONNECTIONS.md와 정렬(소문자).
  **첫 반입 시 생성**(선제 생성 금지 — 존재하는 폴더 목록 = 실제 가동한 도구 목록 = 심사 증거).
  경로 자체가 출처를 증언한다(매니페스트 누락 시의 이중 안전망).
- **② 기록이 입장권**: 착지와 동시에 assets_manifest.json 기록. "나중에 기록"이 아니라
  기록해야 ③으로 간다. 프롬프트 기록 = 재현성(원본이 없어도 재생성 가능). §8 라이선스 훅이 강제 장치.
- **③ 검역**: 기존 reviewer 기준(폴리·원점·스타일·임포트·라이선스) + **확장자 화이트리스트**
  (glb/fbx/png/wav/ogg/mp3만 통과 — 실행파일·스크립트 동봉물은 무조건 반려).
- **④ 파일명 = bom_id 고정**: 소켓 스왑·프리팹 참조가 파일명에 걸려 있다. 재생성은 같은 이름으로
  덮어쓰고, 이력은 git이, 세대는 manifest의 `gen` 카운트가 보관.
- **⑤ 원본 폐기**: 검역 통과 후 _intake 원본 삭제. _intake 잔류물 = 검역 미통과 방치물이라는 신호.
- **GC 연동**: 체크포인트마다 _intake/* 도구별 잔류 확인. `mixamo/` 적체 = 사람 매개 병목,
  `meshy/` 적체 = 검역이 생성 속도를 못 따라감(TOC 감속 신호). 폴더 구조가 병목 진단을 공짜로 준다.
- **부수 효과**: manifest 집계 = 도구별 크레딧 장부(잔량 대비 남은 BOM 판단) + 관문 통과 대장(디렉팅 증거).
